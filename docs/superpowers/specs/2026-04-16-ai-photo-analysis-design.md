# AI Photo Analysis ‚Äî Design Spec

**Date:** 2026-04-16
**Project:** snaplist
**Status:** Approved

---

## Goal

When a seller creates a new listing, they upload photos and click "Analyze Photos." The app sends the images to Gemini 2.0 Flash, which identifies the item and returns a title, description, suggested price, category, and condition. These values pre-populate the listing form inline. The seller reviews, edits as needed, and posts.

This is snaplist's core differentiating feature ‚Äî a listing form that fills itself out.

---

## User Flow

**Step 1 ‚Äî Photos only**
The new listing page shows only a photo upload drop zone and an "Analyze Photos" button. A tip below the upload area reads: *"Upload 2‚Äì3 photos for the most accurate identification."*

**Step 2 ‚Äî Form revealed**
After a successful analysis, the photo strip collapses, a green success banner appears ("Item identified ‚Äî review and edit the details below"), and all form fields are revealed pre-filled. Fields populated by AI show a `‚ú¶ AI` badge. The user edits freely and posts.

**Re-analyze**
A "Re-analyze" button in the photo strip lets the user swap photos and run analysis again. Disabled after 3 attempts per session with the message: *"Maximum analysis attempts reached. Edit the fields manually."*

**ZIP code**
Auto-populated from the user's saved ZIP (fetched from `userSettings` on page load). Falls back to browser geolocation, then manual entry. ZIP is saved to `userSettings` on every successful listing post.

**Error state**
If Gemini cannot identify the item: *"We couldn't identify this item. Try adding more photos or fill in the details manually."* The form fields are revealed so the user can proceed manually.

## AI Layer

### `lib/ai/analyze.ts`

```ts
export interface ListingAnalysis {
  title: string
  description: string
  price: number        // cents
  categorySlug: string // matched to existing categories
  condition: 'new' | 'like_new' | 'good' | 'fair'
}

export async function analyzeListing(
  images: string[],    // base64-encoded
  location: string,    // "City, ST" derived from zip
  categories: string[] // slugs from DB
): Promise<ListingAnalysis>
```

**Prompt design:** Single-turn prompt instructing Gemini to identify the item, write a concise marketplace description, estimate a fair used-market price for the given location, match the category exactly to the provided list, and assess condition from visible wear. Response must be valid JSON matching `ListingAnalysis`.

**Model:** `gemini-2.0-flash` via `@google/generative-ai` SDK.
**API key:** `GOOGLE_AI_API_KEY` env var.

### `lib/ai/pricing.ts`

```ts
export async function getPriceSuggestion(
  item: string,
  location: string
): Promise<number> // cents
```

Today this simply returns the price Gemini already suggested (passed through from `analyzeListing`). When market data is integrated, this function is the only change point ‚Äî callers don't need to change.

## API Route

### `POST /api/analyze`

**Auth:** Clerk session (`auth()` from `@clerk/nextjs/server`). Returns 401 if unauthenticated.

**Rate limiting:** Reuses `checkRateLimit()` from `lib/api.ts` (60 req/min per IP).

**Request:** `multipart/form-data`
- `images` ‚Äî 1‚Äì10 image files (JPEG/PNG, max 5MB each)
- `zip` ‚Äî 5-digit string

**Response (success):**
```json
{
  "title": "Apple MacBook Pro 13\" (2021) ‚Äî M1",
  "description": "...",
  "price": 74900,
  "categorySlug": "electronics",
  "condition": "like_new"
}
```

**Response (error):**
```json
{ "error": "Could not identify item from the provided photos." }
```

**Processing:**
1. Validate image count and file types
2. Resolve zip ‚Üí city/state via `lookupZip()` (already in `lib/geo.ts`)
3. Fetch category slugs from DB
4. Convert images to base64
5. Call `analyzeListing(images, location, categories)`
6. Return structured result

## ZIP Persistence

**On load (`app/listings/new/page.tsx`):**
Calls `getUserSettings()` server action ‚Üí returns `{ zip }` or null. Passed as `defaultZip` prop to the form. If null, the form requests browser geolocation on mount.

**On post (`app/actions/listings.ts` ‚Üí `createListing()`):**
After inserting the listing, upserts `{ clerkUserId, zip, updatedAt }` into `userSettings`. Non-blocking ‚Äî ZIP save failure does not fail the listing creation.

---

## Analysis Attempt Limit

Tracked in React state (`analysisCount`, initialised to 0). The "Analyze Photos" button is disabled when `analysisCount >= 3`. No server-side tracking needed ‚Äî the general rate limiter on `/api/analyze` covers abuse at scale.

## Error Handling

| Scenario | Behaviour |
|---|---|
| Gemini returns unparseable JSON | Catch, return `{ error }` from route |
| Item not identified / low confidence | Gemini instructed to return `{ error: "..." }` in prompt |
| API key missing/invalid | 500 with generic error; do not expose key details |
| Image too large / wrong type | 400 with validation message before calling Gemini |
| ZIP not found | Omit location from prompt; Gemini estimates without it |

## Out of Scope

- Market-data pricing (eBay, PriceCharting) ‚Äî `lib/ai/pricing.ts` is the hook
- API key auth on `/api/analyze` ‚Äî Clerk session only for now
- Analysis on the edit form ‚Äî existing listings keep their data
- Per-user daily analysis quota beyond the 3-per-session client limit
